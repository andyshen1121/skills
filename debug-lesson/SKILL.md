# Debug Lesson — 调试教训沉淀

当你刚刚修复了一个 bug 或解决了一个棘手问题，使用此技能将经验沉淀到 CLAUDE.md 中，避免未来重复踩坑。

## 触发时机

- 用户说"记录这个教训"、"加到调试教训里"、"别再犯这个错误"等
- 用户在修复 bug 后说 `/debug-lesson`
- 也可附带描述：`/debug-lesson Redis 重试导致超时`

## 执行流程

### Step 1: 提取教训内容

从当前对话上下文中提取以下信息（如果上下文不够清晰，向用户确认）：

- **问题现象**：用户遇到了什么错误/异常行为
- **根因分析**：问题的根本原因是什么
- **修复方案**：如何解决的
- **防范规则**：未来应该遵守的规则（这是最关键的部分）

### Step 2: 判断教训级别

根据教训的适用范围，决定写入哪个 CLAUDE.md：

**写入项目级** `./CLAUDE.md` 的情况：
- 与特定框架/库的版本行为相关（如 "Next.js 16 的 params 必须 await"）
- 与项目特定的 API/服务相关（如 "DashScope 的 enable_thinking 互斥"）
- 与项目架构设计相关（如 "Planner 路由必须用 LLM 不能用正则"）
- 与项目使用的特定 SDK 行为相关

**写入用户级** `~/.claude/CLAUDE.md` 的情况：
- 通用开发模式/反模式（如 "流式内容容器必须自动滚动"）
- 跨语言/跨项目适用的经验（如 "placeholder URL 不要重试"）
- 工具链通用问题（如 "Git worktree 清理"）
- Claude Code 自身的使用技巧

**判断原则**：如果你不确定，偏向写入项目级——因为项目级的教训更容易被精确匹配到相关场景。只有真正跨项目通用的教训才写入用户级。

### Step 3: 格式化教训条目

#### 项目级格式

读取项目 CLAUDE.md 中 `## 调试教训` 部分，找到当前最大编号 N，新条目编号为 N+1。

格式必须与现有条目一致：

```markdown
### N+1. 简短标题（一句话概括问题域）
- 问题现象的具体描述
- **关键规则（加粗）**
- 补充说明（如何检测、如何避免）
```

要求：
- 标题简短有力，用"名词 + 问题关键词"格式（如"Redis placeholder URL = 隐性超时炸弹"）
- 防范规则用 **加粗** 标注，这是最重要的信息
- 用 `-` 无序列表，不要用段落式描述
- 不要写"问题："/"解决方案："这类前缀，直接描述
- 保持与现有条目相同的语言风格（中文为主，技术术语用英文）

#### 用户级格式

如果用户级 CLAUDE.md 中还没有 `## 调试教训` 部分，先创建该部分。

格式同上，但条目编号独立计数（与项目级分开）。

### Step 4: 写入文件

1. 使用 Edit 工具将新条目追加到对应 CLAUDE.md 的调试教训部分末尾
2. 不要修改现有条目的内容或编号

### Step 5: 提交并推送

**仅当写入的是项目级 CLAUDE.md 时**执行 git 操作：

```bash
git add CLAUDE.md
git commit -m "docs: add debug lesson N+1 — 简短描述"
git push
```

用户级 `~/.claude/CLAUDE.md` 不在 git 仓库中，无需 commit/push。

### Step 6: 确认输出

向用户输出：

```
已记录调试教训 #N+1：「标题」
写入位置：[项目级/用户级] CLAUDE.md
适用范围：[仅当前项目 / 所有项目通用]
```

## 注意事项

- 不要重复记录已存在的教训。写入前先搜索现有条目是否已覆盖该问题
- 如果现有条目部分覆盖，优先补充到现有条目而非新增
- 一次只记录一个教训，不要合并多个不相关的问题
- 教训的核心价值是**防范规则**，不是问题描述——侧重"以后怎么做"而非"这次出了什么事"
