# 用中学 -- 场景驱动的渐进式教学法

## 核心原理

人在面对一张陌生命令表时，大脑没有锚点，记不住也不知道什么时候该用。
但如果先给一个具体场景（「你有 5 个 PDF 散落在 Downloads 里，想归到一个文件夹」），
大脑立刻有了画面，这时候引入的 2-3 个新操作会自然地挂到这个场景上，形成记忆。

所以这个 skill 的核心规则是：**永远先给场景，再在场景中自然引出知识点。**

## 两种教学模式

### 交互式模式（默认）

每次只教 1-2 课，教完后暂停，等用户反馈再继续。这是默认模式。

流程：
1. 输出第 1 课
2. 课末给出「试一试」练习
3. 问用户：「试完了吗？有问题吗？继续下一课？」
4. 根据用户反馈决定：继续、跳级、深入、还是换方向
5. 重复直到用户说够了

交互式模式下的关键行为：
- 用户说「这个我会」「跳过」「下一个」-- 直接跳到下一课
- 用户提问 -- 先回答问题，再继续课程
- 用户说「这块多讲讲」-- 在当前场景上展开，补充更多操作和技巧
- 用户说「够了」「差不多了」-- 给出日常使用建议，结束教学

### 完整输出模式

当用户明确要求保存、归档、或说「帮我整理一份教程」时，一次性输出全部课程。
此模式下遵循完整的课程路径设计，一口气输出 5-10 课。

判断标准：
- 用户说「保存到 Obsidian」「写一份教程」「整理成文档」→ 完整输出模式
- 其他所有情况 → 交互式模式

## 教学流程

### 第一步：获取素材

在设计课程前，先获取学习对象的一手信息：

- **CLI 工具 / 软件**：用 WebFetch 抓取官方文档的 Quick Start 或 Getting Started 页面
- **编程框架 / 库**：抓取官方教程或 API 概览页
- **概念 / 方法论**：搜索权威来源的入门解释

这一步的目的是让场景设计基于真实的功能和最佳实践，而不是凭想象编造。
如果官方文档无法获取，基于自身知识进行，但要标注「基于通用经验，具体以官方文档为准」。

### 第二步：了解用户起点

快速判断用户的经验水平，优先从对话上下文推断：

- 用户对这个领域完全陌生，还是有相关经验？
- 用户的使用场景是什么？（开发、数据分析、日常使用？）
- 用户的操作系统和环境？

不需要逐条问。如果能从上下文推断就直接开始，只在关键信息缺失时简短问一句。

**跳级规则**：如果用户有相关工具/领域的经验，跳过重叠部分，从差异和增量开始教。

例如：
- 「我 Vim 很熟，教我 Neovim」→ 跳过基础操作，从 Neovim 独有功能（Lua 配置、内置 LSP）切入
- 「我会 SQL，教我 pandas」→ 用 SQL 类比讲 DataFrame 操作，不讲什么是「筛选」「排序」
- 「用过 Docker，想学 K8s」→ 从 Docker 到 K8s 的概念映射开始，不重复容器基础

### 第三步：设计课程路径

把要教的内容拆成课程，遵循以下原则：

1. **第一课必须让用户「打开它，四处看看」** -- 立刻获得「我能用了」的正反馈
2. **每课对应一个真实场景** -- 不是功能分类，而是用户会遇到的真实问题
3. **每课只引入 2-5 个新知识点** -- 超过 5 个就拆课
4. **后面的课建立在前面的基础上** -- 新课复用前面学过的操作，强化记忆
5. **从高频到低频排列** -- 用户最常遇到的场景放前面
6. **最后一课是「不知道怎么办时怎么办」** -- 教用户自助（文档、帮助命令、社区）

课程数量根据学习对象的复杂度灵活调整：
- 简单工具（cat, curl）：2-3 课就够，不要硬凑
- 中等工具（git, docker）：5-7 课
- 复杂系统（K8s, Vim）：8-10 课

### 第四步：写每一课

```
## 第 N 课：[场景标题，用用户视角描述]

**场景**：[一句话描述用户遇到的具体问题]

[具体操作步骤，用第二人称（「你」），边操作边解释]

[如果有多种方法解决同一问题，用「方法 1/方法 2」分列]

> 本课新学的：`xxx` `yyy` `zzz`

**试一试**：[一个具体的小任务，让用户用本课学到的操作自己完成]
```

关键要求：

- **场景必须具体** -- 不要写「管理文件」，要写「把 Downloads 里散落的 5 个 PDF 归到一个文件夹」
- **用用户自己环境中的真实路径和文件** -- 如果知道用户的工作目录，用它来举例
- **操作步骤中穿插解释，不要把解释攒到最后** -- 用户应该在操作的瞬间理解为什么
- **每课末尾用引用块汇总本课新学的知识点** -- 方便日后速查
- **每课末尾必须有「试一试」** -- 一个让用户动手的小任务，巩固刚学的内容

### 第五步：收尾

教程结束后，给出：

1. **日常使用建议** -- 3-5 条最值得养成的习惯，帮助用户从「学完了」过渡到「用起来了」
2. **进阶方向** -- 如果用户想继续深入，下一步应该学什么
3. **如果用户要求保存** -- 按照 save-to-obsidian skill 的流程保存到 Obsidian vault

## 场景设计的技巧

### 好场景 vs 坏场景

| 坏场景（功能分类的马甲） | 好场景（用户脑子里真实的念头） |
|--------|--------|
| 「文件操作」 | 「把散落的 5 个 PDF 归到一个新文件夹」 |
| 「搜索功能」 | 「找一个忘了放哪的文件，只记得名字里有 report」 |
| 「快捷键」 | 「不用退出编辑器，直接在里面跑测试」 |
| 「配置管理」 | 「换了电脑，想把所有设置搬过去」 |
| 「版本控制」 | 「改着改着发现改错了，想回到昨天的版本」 |

### 难度曲线

```
第 1 课：打开，看看，关掉（获得信心）
第 2-3 课：最高频的日常操作（立刻有用）
第 4-6 课：稍复杂但常见的工作流（感受威力）
第 7-8 课：高级技巧和组合操作（进阶提升）
最后一课：自定义和自助（独立成长）
```

### 「试一试」设计原则

好的练习任务：
- 用本课学到的操作就能完成，不需要还没教的知识
- 有明确的完成标准（「你应该看到 X」「文件应该出现在 Y」）
- 和用户的真实环境相关（用他们自己的文件/目录/项目）
- 难度略高于课上演示的操作，需要用户自己组合一下

例如：
- 教了 yazi 的选择和移动 → 「试试把你 Downloads 里最近 3 天下载的图片挑出来，移到一个新文件夹」
- 教了 git commit → 「修改项目里的 README，加一行你的名字，然后提交」
- 教了 pandas 读取 CSV → 「用 pd.read_csv 读取你手头任意一个 CSV 文件，打印前 5 行看看」

## 针对不同类型学习对象的适配

### CLI 工具（如 git, docker, vim, yazi）
- 每个操作给出要输入的命令，用代码块
- 描述执行后屏幕上会看到什么变化
- 如果有危险操作（如删除），明确标注

### GUI 软件（如 IDE, Figma, Excel）
- 用「点击 X → 选择 Y」的格式描述路径
- 如果有快捷键，在操作旁标注，但不要先列快捷键表
- 截图路径如果能生成/获取就用截图

### 编程概念（如设计模式、算法、框架）
- 用类比连接到用户已知的概念
- 先给最小可运行的代码示例，再逐步扩展
- 每个概念都要回答「什么时候该用它」和「不用它会怎样」
- 「试一试」环节给出可以直接跑的代码骨架，让用户填空

### 业务/领域知识（如财务分析、运维流程）
- 从一个具体案例切入，不要从定义开始
- 用数字和实例说话，不要抽象描述

## 禁止事项

1. **禁止在场景之前列出命令/API/快捷键参考表** -- 这是最核心的禁令
2. **禁止一次性倾倒所有功能** -- 交互模式下每次只教 1-2 课
3. **禁止脱离场景解释概念** -- 每个知识点必须长在场景上
4. **禁止使用「首先了解基本概念」开头** -- 直接进入第一个场景
5. **禁止假设用户知道术语** -- 第一次出现的术语在使用时顺带解释
6. **禁止跳过「试一试」环节** -- 每课必须有动手练习
7. **禁止对有经验的用户从零教起** -- 检测到相关经验时必须跳级
