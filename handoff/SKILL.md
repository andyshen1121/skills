# Handoff

解决上下文压缩丢失关键信息的问题。压缩算法按"可重读"判断，会优先删掉 Tool Output 和文件内容，架构决策的理由和踩坑记录也会一起消失。

**两种触发模式：**
- 用户说"写"/"记录"/"交接"/"开新会话" → **写入模式**
- 用户说"读"/"加载"/"看看" → **读取模式**
- 只说"handoff"且无上下文 → 问用户："是要写交接文件，还是读取现有的？"

---

## 写入模式（会话结束前）

按以下结构从当前对话中提取信息，写入当前工作目录的 `HANDOFF.md`：

### 必须提取的内容

**1. 当前任务和完成状态**
- 在做什么，完成到哪一步
- 用一句话说清楚"现在卡在哪里"或"下一步是什么"

**2. 架构决策（带原因）**
- 选了什么方案、为什么选
- 排除了什么方案、为什么排除
- 这是压缩最容易丢失的信息，必须完整保留

**3. 踩过的坑（死路记录）**
- 尝试过但失败的方向，附上失败原因
- 目的：让新 agent 不重复踩坑

**4. 当前文件改动状态**
- 修改了哪些文件
- 每个文件的关键变更是什么（不是全文，是变更点）

**5. 验证状态**
- 测试/验证结果：通过还是失败
- 如果失败，现象是什么

**6. 明确的下一步**
- 新 agent 拿到这个文件后第一件事是什么
- 要足够具体，能直接执行

### 写入格式

```markdown
# HANDOFF

> 更新时间：{YYYY-MM-DD HH:mm} | 工作目录：{cwd}

## 当前任务
{一句话描述在做什么，完成度如何}

## 下一步（新 Agent 从这里开始）
{具体、可执行的下一步操作，不能是模糊描述}

## 架构决策
| 决策 | 选择 | 原因 |
|------|------|------|
| {决策点} | {选择了什么} | {为什么} |

## 死路记录（不要重复尝试）
- **{方案名}**：{尝试结果} → {失败原因}

## 文件改动
- `{文件路径}`：{关键变更描述}

## 验证状态
- {测试/检查项}：{通过/失败} — {现象或日志片段}

## 背景补充
{其他新 Agent 需要知道的上下文，可选}
```

### 写入后

告知用户：
```
HANDOFF.md 已写入 {路径}

建议快速扫一眼，有缺漏直接告诉我补。
确认后开新会话，把这个路径发给新 Claude 即可。
```

---

## 读取模式（新会话开始时）

1. 在当前工作目录查找 `HANDOFF.md`
2. 读取后输出：

```
[HANDOFF 已加载]

任务：{当前任务}
下一步：{明确的下一步操作}
需注意：{死路记录中的关键禁忌，一句话}
文件状态：{改动文件数量和关键变更}
验证：{通过/失败状态}
```

3. 直接问："要开始执行下一步吗？"

**如果文件不存在**：告知用户当前目录无 HANDOFF.md，询问是否需要新建。
